Lachesis Backend Technical Documentation

Generated from current backend code in qtbn_simulator_clean.py, Qtbn_UI.py, qaoa_scenario1.py, vqe_tab.py, qtbn_core.py, and foresight.py.

1) App Overview (same framing as prior doc)

Lachesis is a single-process Streamlit application that combines quantum simulation, market risk analytics, scenario modeling, and AI-assisted explanation. The app behaves like an integrated research platform rather than a classic microservice backend. Most "backend" logic is implemented as Python service functions inside Streamlit modules and executed in-process on user interaction.

What the app is, technically

  • A stateful Python analytics engine using st.session_state as its runtime state store.
  • A computation stack that spans quantum simulation (Qiskit/Aer), Monte Carlo risk modeling, optimization (CVXPY), and NLP/LLM integrations.
  • A role-gated secret/key-resolution system with owner-only key entry and server-managed fallback secrets/env.
  • A file-backed artifact system for snapshots, run logs, calibration history, scenarios, and governance outputs.

What that means in plain terms

The app is one smart control center. It does heavy calculations, remembers your session settings, pulls external data, and can explain results in plain English, all from one dashboard.

2) Backend Topology

2.1 Execution model

  • Runtime model is event-driven Streamlit reruns. Each widget interaction triggers a script rerun against current st.session_state.
  • No standalone API server process is defined in this codebase; backend routines are local function calls.
  • Caching is used selectively via @st.cache_data (e.g., market fetch, returns preprocessing) and functools.lru_cache (simulator/transpile/count pipelines).

2.2 Module map

  • qtbn_simulator_clean.py: primary orchestration module and majority of backend computations.
  • Qtbn_UI.py: auth gate, owner detection, key resolution, voice synthesis utilities, theme and auth UI wrappers.
  • qaoa_scenario1.py: QAOA mini-lab engine, scenario persistence, snapshot export into the broader pipeline.
  • vqe_tab.py: VQE mini-lab, smoke checks, risk budget mapping, trade gate enforcement wrapper and audit trail.
  • qtbn_core.py: standalone minimal QTBN engine abstraction (regime transition forward model).
  • foresight.py: lightweight foresight helpers for count aggregation, KL divergence, and blending.

2.3 Data planes

  • In-memory/session plane: st.session_state keys for all active controls, model state, and transient artifacts.
  • Disk plane: JSON/CSV/TOML files for scenario and model artifacts.
  • Network plane: external providers (OpenAI, Perplexity, yfinance, FRED, SEC EDGAR, Supabase).

3) Security and Identity Backend

3.1 Authentication pipeline

Technical

  • render_auth_gate() in Qtbn_UI.py is the authentication gatekeeper.
  • Primary auth path: Supabase via URL + anon key loaded from env/secrets.
  • Failover path: local auth_users.json credential store with per-user salt and SHA-256 digest.
  • Canonical identity state:
    • auth_ok
    • auth_source
    • auth_email_normalized (trim + lowercase)
    • auth_is_owner (true only if email equals darriusperson@gmail.com)

Layman

The app first tries cloud login. If that fails, it can still log in locally. It standardizes your email and marks the owner account automatically.

3.2 API key resolution and owner-only controls

Technical

  • Central resolver: resolve_api_key(service).
  • Resolver precedence:
    1. Owner session override (Admin input field)
    2. st.secrets
    3. Project-local .streamlit/secrets.toml parse fallback
    4. Environment variable
    5. Empty key
  • Owner session cleanup on logout via clear_auth_session() and clear_session_api_key_overrides().
  • Service map includes OpenAI, FRED, Perplexity, OpenAI-voice, ElevenLabs-voice.

Layman

Only the owner can type keys into the app. Other users can still use features when backend keys are configured, but keys are hidden.

Technical

Layman

Your app preferences are saved, but sensitive keys are intentionally not written there.

5) Dependency Management and Runtime Guardrails

5.1 Optional vs hard dependencies

Technical

  • Hard dependency: Qiskit core/Aer. If missing, app stops with installation instructions.
  • Optional dependencies degrade gracefully:
    • yfinance, fredapi, openai, cvxpy, reportlab, toml, feedparser, nltk, plotly.
  • QAE components imported with new/legacy shim logic and tracked in _QAE_ERRORS.
  • Runtime diagnostics expander reports environment versions and unavailable features.

Layman

If some advanced libraries are missing, the app still runs and shows fallback behavior instead of crashing.

5.2 Browser compatibility patch

Technical

  • Startup includes Safari-specific Streamlit static asset patching and cache-busting for UI compatibility.
  • This is backend-adjacent operational hardening, not core analytics logic.

6) Quantum Backend Engine

6.1 Circuit assembly

Technical

  • Core builders:
    • build_unitary_circuit_1q()
    • build_unitary_circuit() for 1-4 qubits over T0/T1/T2 gate schedule
    • build_measure_circuit_with_noise() for measured execution
  • apply_gate() supports H/X/Y/Z/RX/RY/RZ/S/T with angle scaling in pi units.
  • Conditional CNOT insertions per step and qubit topology (0-1, 1-2, 2-3).

Layman

The backend converts sidebar gate choices into actual quantum circuits that can be run or measured.

6.2 Noise model backend

Technical

  • get_simulator() builds AerSimulator with optional NoiseModel.
  • Noise channels:
    • Single-qubit depolarizing
    • Amplitude damping
    • Phase damping
    • Two-qubit depolarizing for CX
  • Noise parameters can be user-entered or replaced by calibration snapshot values when use_calibrated_noise is true.

Layman

The simulator can mimic real hardware imperfections so results are more realistic.

6.3 Quantum compute caching

Technical

  • QCs registered by hash key via _qc_key() and QC_REGISTRY.
  • Transpilation cached in cached_transpile().
  • Counts execution cached in run_counts_cached() with hashable inputs.

Layman

The app reuses expensive quantum compilation/execution work so repeated runs are faster.

7) Calibration and Quantum Diagnostics Backend

7.1 Tomography and process diagnostics

Technical

  • state_tomography_1q(): basis-rotated measurements estimate Bloch components (X, Y, Z).
  • process_fidelity_basic(): compares ideal vs noisy outputs over basis states and averages fidelity.
  • process_tomography_proxy_1q(): maps prepared axis states through circuit and computes output Bloch vectors.
  • quantum_volume_proxy(): educational proxy using 2**min(width, depth).

Layman

These are health tests for quantum circuits: how accurate they are and how they transform states.

7.2 Auto and Bayesian calibration

Technical

  • auto_calibrate_noise(): heuristic estimates for depolarizing/amplitude/phase rates from simple tomography signatures.
  • bayesian_calibrate_noise():
    • Runs channel experiments
    • Updates Beta priors to posteriors per channel
    • Returns mean + credible intervals via _beta_ci()
  • Randomized benchmarking randomized_benchmarking_1q() fits survival curve model A * p^m + B and computes EPG approximation.
  • Calibration snapshots retained in-session (bounded list).

Layman

The app can estimate how noisy your virtual quantum hardware is, with both quick and statistically richer methods.

8) Market Data and Macro Backend

8.1 Market data ingestion

Technical

  • fetch_market_data():
    • In demo mode: synthetic price generator
    • Otherwise: yfinance download path
    • If remote fetch fails: fallback to synthetic series
  • Input contract: CSV ticker string + lookback days.
  • Output contract: normalized DataFrame (single or multi-index columns).

Layman

If real market data is unavailable, the app generates realistic fake data so analysis can continue.

8.2 Macro data ingestion and fallback

Technical

  • fetch_fred_bundle(api_key) returns CPI/Unemployment/10Y yield.
  • Returns None if no key, no dependency, or fetch failure.
  • simulate_macro_bundle() synthesizes macro values from observed return volatility regime.
  • macro_stress_multiplier() computes clipped stress factor:
    • stress = 1 + 0.40*((u-4)/4) + 0.20*((y10-4)/4)
    • Clipped to [0.8, 1.8]

Layman

The app uses economic indicators when possible, but can estimate macro stress from market behavior if those indicators are unavailable.

9) Risk Engine Backend (VaR/CVaR and metrics)

9.1 Return preprocessing

  • log_return_frame() computes log returns from Adj Close/Close and drops invalid rows.
  • Handles both single-index and multi-index market data frames.

9.2 VaR/CVaR engine

Technical

  • Primary function: monte_carlo_var_cvar(data, horizon_days, sims, alpha).
  • Classical branch:
    • Basket returns = equal-weight mean of asset return series
    • Generates normal draws in chunks for memory control
    • Aggregates horizon-day sums
    • VaR = percentile((1-alpha)*100), CVaR = conditional mean of tail draws
  • QAE branch (optional, guarded by availability + toggle):
    • Builds NormalDistribution circuit over horizon returns
    • Runs Iterative Amplitude Estimation as tail-probability proxy
    • Still reports VaR/CVaR using normal-analytic expressions for consistency
  • Failure behavior: logs exception, returns NaN-safe outputs.

Layman

The backend estimates potential losses by simulating many future scenarios, and can optionally use quantum estimation tools for tail-probability proxying.

9.3 Regime detection and advanced risk metrics

Technical

  • detect_regime() computes 21-day rolling vol annualized and bins into Low/Medium/High volatility.
  • Advanced metrics include Sharpe, Sortino, max drawdown, historical VaR, expected shortfall, skewness, kurtosis.
  • Charting backend in create_comprehensive_financial_charts() provides candlestick/price trends, return distribution with normal fit, rolling vol, and correlation matrix.

Layman

The app classifies the current market mood and computes standard institutional risk/quality indicators.

10) Portfolio Optimization and Backtesting Backend

10.1 Regime-aware optimizer

Technical

  • regime_aware_portfolio_optimizer() uses CVXPY for constrained optimization.
  • Supports:
    • Objective modes: CVaR or variance-style risk objective
    • Long-only and per-asset max-weight constraints
    • Turnover penalty vs previous weights
    • Sector-cap constraints
    • Regime-conditional covariance via kmeans clustering
    • Optional Black-Litterman style views adjustment
  • CVaR objective uses auxiliary variables and alpha-tail constraints over scenario set.
  • Optimization fallback returns equal weights on solver failure.

Layman

This engine chooses portfolio weights while balancing expected return, risk limits, turnover, and diversification constraints.

10.2 Efficient frontier, stress testing, and backtesting

  • plot_efficient_frontier(): sweeps target return and solves minimum variance portfolios.
  • stress_test_portfolio(): compares base vs stressed VaR/CVaR under stress factor scaling.
  • backtest_rebalance(): monthly rebalance simulation with transaction costs, rolling VaR, and drawdown tracking.

11) Sentiment Backend

11.1 RSS + VADER path

Technical

  • analyze_sentiment() pulls Google News RSS for each ticker (up to 10 entries/ticker).
  • Scores each headline with VADER compound sentiment.
  • Risk multiplier formula:
    • multiplier = 1 - 0.5 * avg_score
    • Clipped to [0.5, 1.5]
  • Returns structured payload with titles, links, scores, and provider metadata.

Layman

The backend turns news tone into a number that raises or lowers risk sensitivity.

11.2 Perplexity path

Technical

  • analyze_sentiment_perplexity() sends a strict JSON-only prompt to Perplexity chat completions endpoint.
  • Parses content robustly from multiple response formats and extracts JSON object.
  • Validates and clamps avg_score, per-item scores, and multiplier range.
  • Returns same schema as RSS path for backend interchangeability.

Layman

This path asks a web-aware AI to summarize market sentiment in one structured result format.

13) Foresight Backend

13.1 Baseline and gate evaluation

Technical

  • Baseline fallback model _compute_baseline_fallback() produces:

Layman

13.2 Scenario ranking, sweeps, and fixtures

Technical

  • Sweep manager runs toy what-if perturbations and stores per-run data/metadata in session.
  • Fixtures payload includes baseline/gate/blessing/build/vqe/sweeps/toy-forecast and supports JSON download + disk write.

Layman

The system can compare possible futures, score how close they are, and save those analyses for later review.

14) QAOA Backend (portfolio stance engine)

14.1 Optimization model

Technical

  • Core objective in run_qaoa_portfolio():
    • J(x) = lambda * expected_return - (1 - lambda) * risk
    • energy = -J(x)
    • Binary inclusion vector x over asset universe
  • Classical brute-force path enumerates all bitstrings and selects argmax objective.
  • Qiskit path converts objective into QUBO and uses MinimumEigenOptimizer with QAOA/Sampler where available.
  • Multiple fallback layers route back to classical solver for robustness.

Layman

QAOA backend picks which assets to include by balancing return and risk through a tunable risk-aversion dial.

14.2 Logging, scenarios, and stance export

  • Every run can be logged to CSV with backend type, lambda, bitstring, risk-return stats, and metadata.
  • Saved scenarios persist in JSON and support comparison workflows.
  • export_qaoa_snapshot() writes a compact stance snapshot used by Foresight and Financial tabs.
  • Snapshot loader can upgrade legacy snapshots by recomputing missing expected return/risk/objective and crash metrics.

15) VQE Backend (optimization + governance gate)

15.1 VQE compute pipeline

Technical

  • Supports problem families: toy Hamiltonian, custom Pauli, MaxCut graph, Ising.
  • Attempts real VQE run first (_try_run_real_vqe()), then toy fallback energy path on failure.
  • Transforms energy into risk-budget multiplier (_energy_to_risk_multiplier) in [0.5, 1.5] range for downstream policy control.
  • Smoke evaluation compares run energy to reference source and assigns PASS/WARN/FAIL.

Layman

VQE backend computes an optimization signal and converts it into "how strict should trading limits be".

15.2 Step 8.8 risk gate utilities

Technical

  • build_scaled_risk_limits(): scales configured limit fields by multiplier.
  • apply_risk_gates() decision semantics:
    • REJECTED for VaR/CVaR/leverage breaches
    • CLAMPED for over-notional orders
    • APPROVED otherwise
  • enforce_risk_gate_and_execute() is a choke-point wrapper that only executes on APPROVED/CLAMPED.

Layman

This layer is a strict gatekeeper that can block or shrink risky orders before execution.

15.3 Step 8.9 execution wiring and audit

Technical

  • estimate_order_risk() placeholder computes proxy VaR/CVaR/leverage from notional + volatility + equity.
  • enforce_smoke_policy_and_execute() adds smoke-status policy modes:
    • Off: ignore smoke
    • Moderate: WARN tightens request, FAIL blocks/clamps by policy
    • Strict: WARN treated as FAIL
  • submit_order_through_vqe_gate() orchestrates estimate -> policy -> gate -> execute -> audit pipeline.
  • Audit records are appended to bounded in-memory log and optionally persisted to JSON files.

Layman

Before any trade, backend runs risk checks, applies confidence policy, then logs exactly what happened for traceability.

16) Error Handling and Fallback Design

  • Most external integrations are wrapped with try/except and return typed fallback payloads.
  • Backend prefers continuity of analysis over hard failure.
  • Fallback examples:
    • Supabase -> local auth
    • FRED -> simulated macro
    • yfinance -> synthetic prices
    • OpenAI absent -> local/explanatory stubs
    • real VQE failure -> toy fallback with artifacts preserved

17) Observability and Operational Notes

  • Global logger with informational and error logs across major backend functions.
  • Runtime diagnostics panel surfaces environment versions and missing dependencies.
  • Artifact persistence in golden_build creates operational trail for model quality and trade-gate state.
  • Risk/history logs are bounded in session to avoid unbounded memory growth.

18) Backend Constraints and Engineering Risks

  • Single-process Streamlit architecture limits horizontal scalability and strict API-style separation.
  • Session-state coupling can make deterministic reproducibility sensitive to UI interaction order.
  • Some duplicate helper definitions and large monolithic file sections increase maintenance complexity.
  • Trade risk estimator in Step 8.9 is intentionally placeholder and should be replaced with production VaR/CVaR estimators.
  • Several analytics are toy/prototype approximations by design (explicitly labeled in UI and code).

19) Layman Breakdown by Backend Layer

19.1 Identity and keys

The system knows who is signed in, and only the owner can directly type API keys. Everyone else uses hidden backend keys when available.

19.2 Computation engine

The backend runs quantum simulations, market-risk math, and scenario calculations behind the scenes when you click controls.

19.3 Data sources

It pulls stock, macro, news, and filing data from outside services, and if those fail, it switches to safe fallback data so the app still works.

19.4 Decision controls

QAOA and VQE produce "stance" and "risk strictness" signals. Those signals can tighten or block risky actions.

19.5 Explanations and voice

Lachesis can explain results in human terms and optionally speak them, but still falls back gracefully if voice/LLM providers are unavailable.

19.6 Audit and history

The backend saves important snapshots and logs so decisions can be reviewed later, which is useful for trust and governance.

20) Suggested Backend Hardening Next Steps

  1. Extract monolithic orchestration into service modules (auth, data, risk, quantum, llm, governance).
  2. Add typed schemas (pydantic/dataclasses) for key payloads (risk snapshot, qaoa/vqe artifacts, sentiment response).
  3. Replace placeholder order-risk estimator with production-grade portfolio-aware VaR/CVaR service.
  4. Add unit and integration tests for resolver precedence, fallback behavior, and gate decision matrices.
  5. Introduce explicit backend API layer if multi-user concurrency and deployment scale are priorities.
