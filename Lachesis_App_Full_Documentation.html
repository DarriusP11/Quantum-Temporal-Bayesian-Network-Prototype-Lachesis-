Lachesis / QTBN App - Full Feature Documentation

Generated from the current app codebase (`qtbn_simulator_clean.py`, `Qtbn_UI.py`, `qaoa_scenario1.py`, `vqe_tab.py`).

1) App Overview

This application is a Streamlit-based quantum-finance research workbench. It combines quantum circuit simulation, market risk analytics, scenario intelligence, and LLM-assisted explanations into one interface.

What the app does technically

  • Builds and simulates configurable quantum circuits (statevector, density matrix, noisy measurement, fidelity).
  • Computes Monte Carlo VaR/CVaR, market regime labels, and macro stress overlays (FRED or simulated fallback).
  • Integrates sentiment signals (RSS + VADER or Perplexity live-web sentiment) as a risk multiplier.
  • Supports SEC EDGAR filings retrieval and plain-English LLM summaries.
  • Provides QAOA and VQE mini-labs that export stance/risk signals into the broader decision pipeline.
  • Uses owner-gated API key management so non-owner users never see key inputs.

What this means in plain language

Think of the app as a "control tower" where you can:

  • Experiment with quantum behavior and noise.
  • Translate that into portfolio risk views.
  • Add real-world context (macro data, sentiment, insider filings).
  • Ask Lachesis (the companion explainer) to describe results in understandable terms.

2) Core Architecture and Security Model

2.1 Authentication and Identity

Technical details

  • Auth UI and logic live in `Qtbn_UI.py` (`render_auth_gate`).
  • Primary auth provider is Supabase if `supabase_url` and `supabase_anon_key` exist (env/secrets).
  • If Supabase is unavailable or fails, local failover auth is used via `auth_users.json` with salted SHA-256 password hashes.
  • After successful sign-in, the app stores:
    • `auth_ok`
    • `auth_email_normalized` (lowercased and trimmed)
    • `auth_is_owner` (true only for `darriusperson@gmail.com`)
    • `auth_source` (`supabase` or `local_failover`)

In plain language

The app checks who signed in and labels the owner account based on one exact email. If cloud login is unavailable, local login still works.

2.2 Owner-only API Key Design

Technical details

  • Service key resolution is centralized: `resolve_api_key(service)`.
  • Precedence order:
    1. Owner session override (Admin tab input, owner only)
    2. Streamlit secrets (`st.secrets`)
    3. Local `.streamlit/secrets.toml` fallback parsing
    4. Environment variable fallback
    5. Empty string
  • Service mappings:
    • `openai` -> session `openai_api_key`, secret `openai_api_key`, env `QTBN_OPENAI_API_KEY`
    • `fred` -> session `fred_api_key`, secret `fred_api_key`, env `QTBN_FRED_API_KEY`
    • `perplexity` -> session `perplexity_api_key`, secret `perplexity_api_key`, env `QTBN_PERPLEXITY_API_KEY`
    • `voice_openai` -> session `voice_openai_api_key`, secret `lachesis_voice_api_key`, env `QTBN_LACHESIS_VOICE_API_KEY`
    • `voice_elevenlabs` -> session `voice_elevenlabs_api_key`, secret `elevenlabs_api_key`, env `QTBN_ELEVENLABS_API_KEY`
  • Logout clears auth identity plus API-key related session fields.

In plain language

Only the owner can type API keys in the UI. Everyone else can still use features if the server already has keys configured, but they never see raw key fields.

2.3 Main App Composition

Technical details

  • `qtbn_simulator_clean.py` imports UI/auth helpers from `Qtbn_UI.py`.
  • It safely imports optional modules (VQE tab, QAOA plugin) and degrades gracefully when unavailable.
  • Primary state model is `st.session_state` with utility wrappers `ss_get` / `ss_set`.
  • Persistent user settings are stored in `.qtbn.toml` (non-key settings only).

In plain language

The app is one unified dashboard. It stitches multiple modules together and uses shared session memory so tabs can influence each other.

3) Sidebar Deep Dive (full control surface)

3.1 Account Block

Technical details

  • Shows signed-in normalized email and role badge (`Owner` or `User`).
  • `Log out` calls `clear_auth_session()` and reruns app.

In plain language

This is your identity strip. It confirms who is signed in and lets you end your session quickly.

3.2 Quantum Controls

Technical details

  • Global controls: qubit count (1-4), shots, optional seed and seed value.
  • Time-step style gate scheduling for T0, T1, T2.
  • For each qubit/time-step, user chooses gate from `None/H/X/Y/Z/RX/RY/RZ/S/T` and angle in units of pi.
  • CNOT toggles appear conditionally for multi-qubit setups (0-1, 1-2, 2-3 chains).
  • All values feed `build_unitary_circuit()` and downstream measurement/noise functions.

In plain language

You are building a mini quantum circuit timeline: what operation each qubit gets at each step.

3.3 Noise Controls

Technical details

  • Independent toggles for depolarizing, amplitude damping, phase damping, and CNOT depolarizing noise.
  • Per-step numeric parameters:
    • `p_dep T0/T1/T2`
    • `gamma T0/T1/T2`
    • `lambda T0/T1/T2`
    • `p_CX T0/T1/T2` (if qubits > 1)
  • `Use last calibration as noise` applies calibrated estimates when present.
  • `Auto-calibrate from quick RB/Tomo` runs `auto_calibrate_noise()`, stores confidence and parameters in session.

In plain language

You can inject realistic "hardware imperfections" and optionally auto-fit them from quick calibration experiments.

3.4 Finance / Data Controls

Technical details

  • Ticker input, lookback days, and `Fetch Market Data` button call `fetch_market_data()`.
  • Portfolio value controls support three dollar basis modes:
    • Portfolio total
    • Per-share
    • Position value (shares * price)
  • Risk controls: confidence level alpha, horizon days, MC simulation count, volatility threshold, macro stress toggle.
  • Displays FRED key status only (`Configured/Missing`), never raw key text.
  • Demo mode toggle enables safer synthetic fallback behavior.

In plain language

This is where you pick market data and define how risk should be measured (how far out, how strict, and under what stress assumptions).

3.5 Persistence Controls

Technical details

  • `Save settings` writes selected state to `.qtbn.toml` via `save_persistent_settings()`.
  • `Load settings` restores from `.qtbn.toml` via `load_persistent_settings()`.
  • Any key ending in `_api_key` is ignored on load and excluded from save logic.

In plain language

You can save your non-sensitive app setup and reload it later. API keys are intentionally not persisted there.

4) Tab-by-Tab Documentation (Statevector through VQE)

4.1 Statevector Tab

Technical details

  • Runs `Statevector.from_instruction(build_unitary_circuit())`.
  • For 1 qubit: outputs probability of `|0>` and `|1>`.
  • For multiple qubits: computes probabilities over all basis states and charts them (Plotly bar if available).

In plain language

This tab shows where the quantum system "wants to be" before you measure it.

4.2 Reduced States Tab

Technical details

  • Only active for 2-qubit mode.
  • Builds noisy unitary, runs density-matrix simulator, computes partial traces for each qubit.
  • Calculates Bloch vectors from Pauli X/Y/Z expectations and renders dual Bloch sphere visualization if Plotly exists.

In plain language

This isolates each qubit's individual state so you can see how entanglement/noise changed each one.

4.3 Measurement Tab

Technical details

  • Builds ideal measured circuit and noisy measured circuit.
  • Uses transpile/count caching to reduce repeated runtime cost.
  • Computes distribution distance via total variation and reports robustness as `1 - TV`.
  • Displays ideal vs noisy counts in grouped chart.

In plain language

This compares what results you would get in a perfect world versus a noisy real-world execution.

4.4 Fidelity and Export Tab

Technical details

  • Computes global state fidelity between ideal state and noisy density matrix (`state_fidelity`).
  • Includes executive PDF generation/download hook (`generate_executive_report`).

In plain language

This is your overall quality score for the circuit under current noise settings, plus report export.

4.5 Presets Tab

Technical details

  • Preset buttons apply hard-coded configurations:
    • Bell prep (H -> CX)
    • Dephasing stress
    • Amplitude relaxation
  • Scenario selector maps named scenario templates to circuit/noise settings (`apply_scenario_preset`).

In plain language

One-click starting points so you can instantly switch to common experiment setups.

4.6 Present Scenarios Tab

Technical details

  • Runs a quick ideal-vs-noisy counts comparison for the current active configuration.
  • Reports robustness and count chart, similar to Measurement tab but packaged as a rapid scenario check.

In plain language

A fast health check to see how stable the current scenario is.

4.7 Foresight Tab (Analytical Foresight + Golden Build)

Technical details

  • Initializes dedicated foresight state namespace (`fx_*`) with threshold defaults.
  • Golden Build controls:
    • Directory selection (`fx_gb_dir_raw` -> derived `fx_gb_dir`)
    • Core stability thresholds (avg_std, uncertainty)
    • Blessing thresholds (TV, KL)
    • Blessed fixture selection from JSON files
    • Enforcement toggle to disable sweeps on FAIL
  • Baseline engine:
    • Multi-seed fallback baseline distribution (`mean_p`, `avg_std`, `uncertainty_penalty`)
    • Core gate status evaluation (PASS/WARN/FAIL)
    • Scenario ranking by KL/TV against active scenario set
  • Blessing gate compares baseline against selected fixture distribution.
  • Build status merge combines three components: Core Gate + Blessing + VQE Smoke.
  • Fixtures export includes baseline/gate/blessing/vqe/sweeps/forecast and supports download + disk write.
  • VQE bridge write buttons persist vqe smoke/snapshot artifacts into golden build folder.
  • Action panel computes stance, P(risk-on), uncertainty, and confidence proxy.
  • Sweep manager supports what-if sweep generation, session storage, CSV upload/compare, and CSV export.
  • Repo scenarios section exposes reload/save placeholders and disk scenario visibility.
  • QAOA prior bridge reads snapshot-style persona/lambda/return/crash/allocation values from session.
  • Toy QTBN regime forecast produces calm/stressed/crisis trajectories and line chart.
  • Persona lens narration provides CIO/Risk Officer/Retail interpretation text.
  • Debug expander prints full current foresight state package.

In plain language

This tab is the decision hub. It checks if your model setup is stable, compares it to known good snapshots, merges VQE confidence signals, runs what-if sweeps, and translates probabilities into actionable stance language.

4.8 Financial Analysis Tab

Technical details

  • Requires fetched market data; otherwise prompts user to fetch via sidebar.
  • Optional screenshot importer:
    • Uploads image
    • Calls OpenAI Responses API directly (`/v1/responses`) with strict JSON schema prompt
    • Extracts ticker/shares/market value/diversity/price
    • Can auto-apply inferred portfolio value and ticker into app inputs
  • Position sizing supports two modes:
    • Portfolio dollar input -> implied shares
    • Shares input -> implied portfolio dollars
  • Risk computation:
    • Log returns + advanced metrics
    • Monte Carlo VaR/CVaR (`monte_carlo_var_cvar`)
    • Optional QAE toggle if dependencies available
    • Sentiment multiplier applied to VaR/CVaR
    • Regime detection using volatility threshold
  • Macro stress:
    • Tries live FRED bundle with resolved key
    • Falls back to simulated macro if unavailable
    • Applies stress factor to risk outputs when enabled
  • QAOA stance linkage:
    • Loads `qaoa_snapshot.json`
    • Maps persona/crash index to QTBN prior regime and risk-on prior
    • Displays QAOA vs market regime alignment diagnostics
  • Narrative and explainability:
    • Generates deterministic risk narrative block
    • Lachesis explainer supports OpenAI or local fallback answer logic
    • Optional voice synthesis via selected provider
  • History and exports:
    • Stores risk snapshots in `risk_history` (bounded)
    • Downloads narrative as TXT and JSON
    • Exports full history CSV/JSON
    • Supports current-vs-past snapshot comparison panel with deltas and chart

In plain language

This tab turns raw market data into practical downside estimates (bad day and tail-loss scenarios), lets you stress-test for macro shocks, and gives role-specific explanations for decision makers.

4.9 Insider Trading Tab

Technical details

  • Fetches SEC EDGAR submissions from `data.sec.gov`.
  • Supports ticker-to-CIK resolution via SEC company tickers endpoint.
  • Requires SEC-compliant User-Agent with contact info.
  • Filters forms (3/4/5), limits number of filings, builds direct archive URLs.
  • Caches ticker map and submissions to reduce repeated network cost.
  • Lachesis summary mode:
    • OpenAI-driven summary if key configured and enabled
    • Local fallback summary otherwise
    • Optional voice synthesis output

In plain language

You can pull insider filing records and have Lachesis explain what those filings might signal, with clear limitations.

4.10 Lachesis Guide Tab

Technical details

  • Loads latest QAOA snapshot and maps it to QTBN priors (regime, risk-on, drift).
  • Shows status metrics for QAOA stance + QTBN prior + QAE/macro toggles.
  • Includes pipeline explainer (QAOA -> QTBN priors -> Financial Analysis).
  • Provides interactive Q&A using full app context via `_studio_run_openai_chat`.
  • Displays noise-aware suggestions.
  • Includes QTBN forecast prototype with selectable horizons and JSON export.

In plain language

This is the app's "translation layer" that explains how all components connect and what your current setup implies.

4.11 Advanced Quantum Tab

Technical details

  • QAOA-QTBN bridge inspector panel.
  • State tomography (1-qubit) with Bloch vector reconstruction and visualization.
  • Process fidelity proxy for selected single-qubit gates.
  • Quantum Volume proxy metrics from current circuit width/depth.
  • Least-squares noise auto-calibration with confidence score.
  • Bayesian calibration with user-provided prior JSON and posterior intervals.
  • Calibration snapshots: select/apply/clear historical calibration sets.
  • Randomized Benchmarking (1-qubit) with fit and estimated per-gate error.
  • Process tomography proxy mapping axis inputs to output Bloch vectors.
  • Noise-aware circuit suggestions generator.

In plain language

This is the diagnostics lab. It helps you understand whether your quantum setup is trustworthy and how to improve it.

4.12 Toy QAOA Tab

Technical details

The tab has two layers:

  • Layer A: Full QAOA mini-lab (`qaoa_scenario1.render_qaoa_tab`)
    • Backend mode selection (classical, sampler stub, aer stub).
    • Portfolio universe selection (toy 3-asset vs Lachesis 5-asset benchmark).
    • Regime overlays (baseline, bull, bear, shock) modify mu/cov.
    • Hyperparameters: depth, shots, risk-aversion lambda.
    • Persona buttons set lambda presets (Conservative/Balanced/Aggressive).
    • Run pipeline logs both classical baseline and selected backend results.
    • Result cards: energy, return, variance, objective, bitstring, selected assets.
    • Lachesis narrative generation from current run.
    • Export stance snapshot to `qaoa_snapshot.json` for foresight/risk tabs.
    • Uplift analytics vs classical across logged runs and overlapping lambdas.
    • Lambda sweep plotting and interpretation.
    • Portfolio summary including mu/sigma/correlation and asset class mix.
    • Scenario library: save/load scenarios, compare two scenarios, storyline chart.
  • Layer B: Additional in-file stub controls (`qtbn_simulator_clean.py`)
    • Displays toy portfolio config JSON.
    • Runs local stub optimizer and stores `qaoa_last_result`.
    • Computes crash index heuristic and exports another stance snapshot flow.
    • Maps lambda to persona and crash index to regime for QTBN handoff.

In plain language

This tab is where the app builds a "risk stance" using QAOA-like portfolio selection logic and passes that stance into broader risk forecasting.

  • Shows scored headlines and copy-friendly links.

  • Lachesis sentiment explainer panel supports pasted links/headlines, optional OpenAI mode, voice playback, and local fallback.

In plain language

This tab turns market news tone into a simple dial that can increase or decrease downside risk estimates.

4.15 VQE Tab (from `vqe_tab.py`)

Technical details

  • Isolated mini-lab with feature flag `VQE_TAB_ENABLED` and dedicated state keys (`vqe_*`).
  • Run controls:
    • Try real Qiskit VQE toggle
    • Backend choice (Estimator / AerEstimator)
    • Seed, ansatz, optimizer, reps, qubits, max iterations
  • Smoke check controls:
    • Reference source (latest snapshot, manual, exact bruteforce, none)
    • PASS/WARN thresholds based on absolute energy delta
    • `exact_max_n` guardrail for bruteforce feasibility
  • Problem modes:
    • Toy Hamiltonian
    • Custom Pauli Hamiltonian
    • MaxCut graph (with graph generator, weighted edges, visualization, Hamiltonian preview)
    • Ising (h vector + J couplings)
  • Decode step:
    • Attempts to decode bitstring distribution from optimized VQE parameters
    • Supports auto/statevector/aer/probability sampling modes
    • Reports expected and best decoded objective plus top-K outcomes
  • Execution flow:
    • Build problem Hamiltonian
    • Try real VQE; fallback to toy energy model if unavailable
    • Convert energy to risk-budget multiplier [0.5, 1.5]
    • Compute smoke object (PASS/WARN/FAIL)
    • Persist `vqe_snapshot` and `vqe_smoke` in session and optionally on disk
    • Save artifacts to `golden_build/vqe_snapshot_*.json` and `vqe_smoke_*.json`
  • Step 8.8 risk gate bridge:
    • Scales base risk limits by current VQE multiplier
    • Stores `vqe_scaled_risk_limits`
    • Simulates gate decisions (APPROVED/CLAMPED/REJECTED)
  • Step 8.9 execution wiring:
    • Pipeline pattern: estimate risk -> smoke policy -> risk gate -> execute -> audit
    • Supports smoke policy modes (Moderate/Strict/Off), WARN tighten factor, FAIL block/clamp behavior
    • `submit_order_through_vqe_gate` acts as single integration wrapper for real trade buttons
    • Maintains bounded `trade_audit_log` and optional audit JSON disk writes
  • Diagnostics/export:
    • Convergence view from callback history
    • Ansatz draw preview
    • Snapshot JSON display/download
    • Load latest snapshot/smoke from disk into session
    • Current bridge signal viewer (`vqe_risk_signal`)
    • Trade audit log viewer and clear action

In plain language

The VQE tab is the advanced optimization and risk-gating engine. It transforms VQE outcomes into stricter or looser trading limits and gives you a full simulated approval pipeline before real execution.

5) Cross-Tab Data Flows

5.1 QAOA -> QTBN/Financial/Foresight

Technical details

  • QAOA exports stance to `qaoa_snapshot.json`.
  • Foresight/Financial/Guide load this snapshot to seed priors: regime, risk-on probability, drift, persona.

In plain language

Portfolio stance discovered in QAOA becomes the starting mindset for downstream risk analysis tabs.

5.2 Sentiment -> Financial Risk

Technical details

  • Sentiment tab writes `sentiment_multiplier`.
  • Financial tab multiplies VaR/CVaR by this factor before dollar conversion.

In plain language

News mood can make estimated risk more conservative or more relaxed.

5.3 VQE -> Foresight and Trade Gate

Technical details

  • VQE writes `vqe_snapshot`, `vqe_smoke`, `vqe_risk_signal`, and scaled limits.
  • Foresight merged build status includes VQE smoke status.
  • Main VQE trade block calls `submit_order_through_vqe_gate` to enforce limits before execution.

In plain language

VQE acts like a risk governor. It can tighten what trades are allowed if model confidence drops.

6) Files, Artifacts, and Persistence

6.1 Runtime/User Data Files

  • `auth_users.json` - local failover auth store (salted hashes).
  • `.qtbn.toml` - persisted non-sensitive UI settings.
  • `qaoa_snapshot.json` - latest QAOA stance snapshot.
  • `qaoa_runs_log.csv` - QAOA run log history.
  • `qaoa_scenarios.json` - saved QAOA scenarios.
  • `scenarios.json` - repository scenario store for foresight hooks.
  • `golden_build/*.json` - VQE snapshots, smoke results, fixtures, and bridge artifacts.

6.2 API and External Integrations

  • OpenAI: chat/explainer, screenshot extraction, OpenAI TTS.
  • Perplexity: live-web sentiment analysis.
  • FRED: macro indicators (CPI, unemployment, 10Y yield).
  • SEC EDGAR: insider filings feed.
  • Supabase: primary auth provider.
  • Qiskit stack: optional quantum algorithm/simulation components.

6.3 Failure/Fallback Strategy

  • Missing API keys -> user-friendly "missing/configure" status and local fallback where available.
  • Missing optional dependencies -> disables related features and surfaces warnings.
  • Network/API failures -> catches exceptions and falls back to synthetic/local alternatives when designed.
  • VQE real run failure -> toy fallback still emits usable artifacts.
  • FRED unavailable -> simulated macro bundle.
  • Supabase unavailable -> local auth store.

7) Practical User Journeys

7.1 Quantum-to-Risk Workflow

  • Configure circuit and noise in sidebar.
  • Validate with Statevector/Measurement/Fidelity.
  • Run QAOA stance and export snapshot.
  • Open Foresight to see prior mapping and build status.
  • Run Financial Analysis for VaR/CVaR and narrative output.

7.2 Risk Governance Workflow (VQE)

  • Run VQE and generate smoke/snapshot artifacts.
  • Review multiplier and scaled limits in Step 8.8.
  • Validate full submit path in Step 8.9 with audit logs.
  • Use same wrapper in production trade button integration.

7.3 Owner Operations Workflow

  • Sign in as owner email.
  • Open Admin tab and set session key overrides as needed.
  • Verify status captions across tabs show configured state.
  • Log out to clear identity and session key overrides.

9) Final Summary (Layman)

Lachesis is a unified quantum + finance analysis app. The sidebar configures your experiment and risk model. Tabs progressively move from raw quantum behavior, to scenario and market risk views, to explainability and governance. QAOA provides stance priors, sentiment adjusts risk intensity, and VQE provides a formal risk gate path for execution control. Owner-only Admin controls keep API key entry private while still allowing non-owner users to operate the app through backend-managed credentials.
